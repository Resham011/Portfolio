class Solution {
  public:
    bool dfs(int node, vector<int>& parent, vector<int>& vis, unordered_map<int, vector<int>>& graph){
        vis[node] = true;

        for(auto& nbr : graph[node]){
            if(!vis[nbr]){
                parent[nbr] = node;
                if(dfs(nbr, parent, vis, graph)) return true;
            } else if(parent[node] != nbr){
                // If neighbor is visited and not the parent => cycle
                return true;
            }
        }
        return false;
    }

    int isTree(int n, int m, vector<vector<int>> &edges) {
        if(m != n - 1) return 0; // Early check: Tree must have exactly n - 1 edges

        unordered_map<int, vector<int>> graph;
        for(auto& edge : edges){
            int u = edge[0];
            int v = edge[1];
            graph[u].push_back(v);
            graph[v].push_back(u);
        }

        vector<int> parent(n, -1);
        vector<int> vis(n, 0);

        // Start DFS from node 0
        if(dfs(0, parent, vis, graph)) return 0;

        // Check if the graph is connected (i.e., all nodes visited)
        for(int i = 0; i < n; i++){
            if(!vis[i]) return 0;
        }

        return 1; // It's a tree
    }
};

🔍 Key Tree Conditions (for undirected graph)

  • Must be connected: All nodes are reachable from any starting node.

  • Must be acyclic: No cycles.

  • Must have exactly n - 1 edges if n nodes → already a quick tree check.

Excellent question — understanding why topological order is possible in a DAG (Directed Acyclic Graph) is a core part of many graph algorithms.

✅ TL;DR:

Topological order is possible in a DAG because there are no cycles, which means we can always find at least one node with no incoming edges (in-degree = 0), and place it first.

📌 Let’s break it down:

🔹 What is Topological Order?

A topological ordering of a directed graph is a linear ordering of its nodes such that:

For every edge u → v, node u comes before v in the ordering.

It only makes sense for directed graphs, and only if the graph is acyclic (no cycles).

🔍 Why Cycles Prevent Topological Ordering

If there’s a cycle like:

css
A → B → C → A

You can't place A before B and C, because you’d also need B before C, and C before A. This creates a contradiction.

🛑 Conclusion: Cycles make it impossible to define any valid linear order.

✅ Why It Works in a DAG

In a DAG, no such cycles exist. So:

🔸 Property 1: There is always at least one node with in-degree 0

That is, at least one node has no dependencies. Why?

  • Suppose every node had in-degree ≥ 1.

  • That would mean there's a path back to every node ⇒ cycle ⇒ contradicts the definition of a DAG.

So at least one source node always exists.

🔸 Property 2: You can remove nodes one by one

  • Remove the node with in-degree 0

  • Remove its outgoing edges

  • Repeat — this is what Kahn’s Algorithm does

  • Or use DFS post-order to reverse finish times

Eventually, all nodes are removed in a valid topological order.

🧠 Visual Example (DAG)

css
A → B → D
↓
C

  • Possible topological order: A, C, B, D

  • All edges respect the rule: A comes before B and C; B before D

🧠 Why It Always Exists in DAGs

Theorem: Every DAG has at least one topological ordering.

Proof Sketch:

  1. Base case: empty DAG → trivially ordered.

  2. Inductive step: remove a node with in-degree 0 (always exists), reduce the graph.

  3. Recurse → add node to front of result → valid ordering.

✅ What Does DFS in Kosaraju Actually Do?

In Kosaraju's algorithm, the first DFS does not compute a topological sort in the strict sense.

Instead, it records finishing times of nodes — which gives us a post-ordering. This post-ordering helps us identify which components are logically "upstream" in the SCC DAG.

🔹 Key Point:

The DFS in Kosaraju is used to identify the order in which to process nodes on the transposed graph so that each DFS on the reversed graph captures one complete SCC.

🧠 What does "finishing time" mean?

  • In DFS, when a node finishes (i.e., all its neighbors are explored), we can push it onto a stack.

  • Nodes that finish later go on top of the stack.

  • These nodes are likely to be entry points to larger components when the graph is transposed.

🔁 Why does this help in SCC detection?

🔸 Imagine the SCC DAG (compressed components):

css
A → B → C

Where A, B, C are SCCs, not individual nodes.

  • When you run DFS on the original graph, you'll finish C first, then B, then A.

  • So, the post-order will look like: C, B, A (top of stack: C).

Now you transpose the graph. The edges become:

css
C → B → A

If you process C first on the transposed graph, it won't reach B or A — so you get just C as one SCC.

Then you process B → get only B as SCC. Then A.

✅ Each DFS on the transposed graph picks out exactly one SCC, in correct order.

📌 What the First DFS Really Provides:

StepPurpose
First DFSRecords post-ordering (finishing time) of nodes
Stack built from DFSGives nodes in an order to find source SCCs first in the DAG
Not a topological sortBecause the original graph may have cycles — but the SCC DAG is a DAG

📦 Final Analogy:

Think of SCCs as "chunks" of cycles. The first DFS doesn’t break cycles — it prepares an order that, when the graph is reversed, allows us to uncover each SCC one by one — from leaves to roots in the SCC DAG.
